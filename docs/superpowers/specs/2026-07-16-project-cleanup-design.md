# Project Cleanup Design

## Goal

Привести проект в более чистое состояние без изменения benchmark workflow:
- оставить рабочий `.env.example`;
- удалить явные лишние файлы;
- сократить накопленные результаты прогонов;
- зафиксировать понятный статус `frameworks/kag` как внешнего vendored checkout.

## Scope

В область изменений входят:
- обновление `.env.example`;
- удаление безопасно-лишних файлов в корне проекта;
- очистка старых run/smoke результатов в `results/`;
- короткое документирование статуса `frameworks/kag`.

В область изменений не входят:
- изменение upstream-кода в `frameworks/kag`;
- изменение CLI;
- изменение логики benchmark execution;
- перевод `frameworks/kag` на submodule/subtree в рамках этой задачи.

## Requirements

- В корне должен остаться понятный шаблон `.env.example`, соответствующий текущей env-driven конфигурации.
- В `results/` должен остаться один последний обычный run и один последний smoke-run на каждый framework, если такие запуски существуют.
- Явно лишние локальные файлы, не являющиеся частью проекта, должны быть удалены.
- `frameworks/kag` должен быть задокументирован как внешний исходный код, используемый проектом локально.

## Design

### 1. `.env.example`

`.env.example` должен содержать только актуальные настройки:
- `OPENAI_BASE_URL`
- `OPENAI_API_KEY`
- `OPENAI_MODEL`
- `OPENAI_EMBEDDING_MODEL`

Шаблон должен быть пригоден и для локального OpenAI-compatible запуска, и для OpenAI API, но в самом файле нужно оставить один ясный базовый вариант с понятными комментариями.

### 2. File Cleanup

Удаляются только безопасные артефакты, не являющиеся исходным кодом или нужной документацией. В первую очередь это:
- одиночные временные лог-файлы в корне проекта;
- лишние служебные файлы, не используемые workflow.

Удаление должно быть консервативным: если файл выглядит как пользовательский артефакт с возможной ценностью, он не удаляется автоматически.

### 3. Results Retention Policy

Для каждого framework сохраняется:
- один последний обычный run;
- один последний smoke-run.

Если для framework есть только smoke или только обычный run, сохраняется последний существующий экземпляр этого типа.

Все более старые результаты удаляются.

Определение “последний” выполняется по timestamp в имени run directory, который уже входит в существующий формат run id.

### 4. `frameworks/kag`

`frameworks/kag` не переносится и не переписывается.

Нужно:
- проверить, является ли каталог отдельным git checkout;
- задокументировать, что это vendored external framework source tree;
- явно указать, что benchmark использует его локально и не считает его внутренним кодом `research_bench`.

Это делает структуру проекта понятнее без рискованной миграции способа хранения внешних зависимостей.

### 5. Documentation

README должен объяснять:
- зачем нужен `.env.example`;
- что старые результаты можно периодически чистить;
- что `frameworks/kag` хранится как внешний framework source tree.

Формулировки должны быть короткими и техническими.

## Risks and Constraints

- Нельзя удалять последний полезный run по framework.
- Нельзя затронуть upstream-содержимое `frameworks/kag`.
- Нельзя удалять `.env`, так как это локальная пользовательская конфигурация.
- Нельзя предполагать, что любой untracked файл безопасно удалить; чистка должна быть точечной.

## Recommendation

Рекомендуемый порядок выполнения:
- сначала обновить `.env.example`;
- затем зафиксировать список сохраняемых run/smoke директорий;
- затем удалить старые результаты и явные мусорные файлы;
- затем обновить README с пояснением про `frameworks/kag`.

Это даёт контролируемую и обратимо-понятную уборку без изменения рабочего кода.
