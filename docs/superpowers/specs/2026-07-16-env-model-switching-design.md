# Env-Driven Model Switching Design

## Goal

Сделать так, чтобы переключение между локальной OpenAI-compatible моделью и OpenAI API выполнялось изменением только `.env`, без ручной правки кода, CLI или framework-specific конфигов.

## Scope

Изменения должны сохранить текущее поведение benchmark workflow и не переписывать проект заново. Публичный CLI, `experiment.py`, существующие команды и структура артефактов остаются прежними.

В область изменений входят:
- единый runtime-конфиг для LLM и embeddings;
- адаптация `check`, `ragas`, `LightRAG`, `KAG` и `MSGraphRAG` к этому конфигу;
- документация по переменным окружения;
- тесты на чтение env и применение конфигурации.

В область изменений не входят:
- изменение upstream-кода в `frameworks/kag`;
- изменение публичного интерфейса CLI;
- изменение формата результатов benchmark.

## Requirements

Проект должен использовать единый набор переменных окружения:
- `OPENAI_BASE_URL`
- `OPENAI_API_KEY`
- `OPENAI_MODEL`
- `OPENAI_EMBEDDING_MODEL`

Если переменные не заданы, проект должен продолжать работать с текущими локальными дефолтами, чтобы не ломать существующий локальный сценарий.

При наличии `.env` все стадии benchmark должны брать настройки только из общего runtime-конфига, а не из локальных hardcoded значений.

## Design

### 1. Central Runtime Config

Добавляется небольшой модуль конфигурации в `research_bench`, который:
- читает переменные окружения;
- подставляет локальные дефолты, если env не задан;
- возвращает единый объект настроек для chat model и embedding model;
- предоставляет явные свойства для `base_url`, `api_key`, `model`, `embedding_model`.

Этот модуль становится единственным источником правды для модельных настроек внутри проекта.

### 2. Configuration Flow

Поток конфигурации после изменения:

`ENV/.env` -> `research_bench runtime config` -> workflow/adapters/checks -> framework execution

Это убирает знание о конкретном провайдере из бизнес-логики и делает переключение модели операцией изменения конфигурации, а не кода.

### 3. Integration Points

Общий runtime-конфиг должен использоваться в следующих местах:
- `research_bench/workflows/checks.py`
- `research_bench/workflows/ragas_verification.py`
- `research_bench/adapters/lightrag.py`
- `research_bench/adapters/kag.py`
- `frameworks/msgraphrag/settings.yaml` через существующий workflow подстановки или генерации
- связанные тесты и документация

Для `KAG` конфиг должен продолжать формировать runtime-настройки поверх существующего шаблона, но chat/embedding значения должны приходить из общего конфига, а не из захардкоженных адресов и моделей.

Для `MSGraphRAG` нельзя рассчитывать на ручное редактирование `settings.yaml` перед запуском; workflow должен сам подставлять значения из общего runtime-конфига.

### 4. Defaults

Локальные дефолты сохраняются как fallback:
- `OPENAI_BASE_URL=http://127.0.0.1:8080/v1`
- `OPENAI_API_KEY=local`
- `OPENAI_MODEL=/models/Qwen3.5-35B-A3B-Q4_K_M.gguf`
- `OPENAI_EMBEDDING_MODEL=multilingual-e5-large`

Это даёт обратную совместимость с текущим локальным запуском.

### 5. Check Semantics

`python experiment.py check` должен валидировать именно активную конфигурацию из env:
- проверять доступность chat endpoint по текущему `OPENAI_BASE_URL`;
- проверять chat completion по текущей `OPENAI_MODEL`;
- проверять embedding endpoint через тот же `OPENAI_BASE_URL`, используя `OPENAI_EMBEDDING_MODEL`, если провайдер общий;
- не считать локальные `127.0.0.1` обязательными, если пользователь переключился на внешний API.

Если текущий провайдер не поддерживает embeddings через ту же конфигурацию или модель задана неверно, check должен падать по фактической конфигурации пользователя, а не по старым локальным ожиданиям.

### 6. Documentation

Нужно обновить документацию так, чтобы было понятно:
- какие env-переменные используются;
- какие значения можно поставить для локального сервера;
- какие значения можно поставить для OpenAI API;
- что достаточно поменять только `.env`, чтобы переключить провайдера.

README и связанные developer docs должны описывать именно новый единый путь конфигурации.

### 7. Testing

Нужно покрыть:
- чтение env-переменных;
- использование дефолтов при отсутствии env;
- подстановку значений в `check`;
- подстановку значений в `ragas`;
- подстановку значений в `LightRAG`;
- подстановку значений в KAG runtime config;
- подстановку значений в MSGraphRAG runtime settings.

Тесты должны доказывать, что достаточно изменить env, а код и workflow остаются неизменными.

## Recommended User Experience

Пользовательский сценарий должен выглядеть так:

1. Открыть `.env`
2. Поменять `OPENAI_BASE_URL`, `OPENAI_API_KEY`, `OPENAI_MODEL`, `OPENAI_EMBEDDING_MODEL`
3. Выполнить `python experiment.py check`
4. Запустить `python experiment.py smoke <framework>` или другой workflow

Без дополнительных изменений в коде.

## Risks and Constraints

- Некоторые framework-specific integration points сейчас ожидают разные chat и embedding endpoints; при реализации важно не привязать новый конфиг к предположению, что всегда существует отдельный embedding сервер.
- `MSGraphRAG` использует свой settings-файл, поэтому изменение должно идти через существующий runtime workflow, а не через ручную постоянную замену upstream-конфига.
- Нужно сохранить совместимость с локальным окружением, иначе проект станет сложнее запускать в текущем режиме.

## Recommendation

Рекомендуемый путь реализации:
- сначала ввести единый runtime-конфиг;
- затем перевести на него `check` и `ragas`;
- затем адаптеры `LightRAG` и `KAG`;
- затем runtime-подстановку для `MSGraphRAG`;
- в конце обновить тесты и документацию.

Это даёт минимальный риск и позволяет верифицировать переход поэтапно.
